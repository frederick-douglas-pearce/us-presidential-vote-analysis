## E8: Internal API

**Issue title:** Epic: internal API over the joined EC+PV dataset
**Labels:** `epic:internal-api`, `enhancement`

**Body:**

### Summary

Expose the cleanly joined EC+PV dataset through an internal HTTP API. Per the roadmap
(M3) and **D002**, the API is a first-class standalone deliverable; the **MVP bar is "it
powers our app."** This epic delivers that MVP surface over the **already-shipped
`ec_pv_redistributable` join view** (E6 / #63), serving only redistributable rows (MIT /
CC0, D016) so the internal MVP and the eventual public graduation are the same surface with
no rework.

**Locked design decisions** (made with Fred before this scoping round ‚Äî spec to them, do
not re-open; proposed as D028‚ÄìD032 below for the record):

1. **FastAPI (REST), not GraphQL.** Plain REST GETs so a CDN can cache responses, and to
   get **auto-generated OpenAPI / Swagger docs** ‚Äî a public-facing asset when the API is
   later advertised (e.g. to the MIT Election Lab).
2. **Serving store = a read-only embedded snapshot** (SQLite file or static JSON ‚Äî
   architect's call) materialized from `ec_pv_redistributable`. **Postgres stays the
   LOCAL warehouse / source of truth; the API has NO live-DB dependency in production.**
   This is the core cost/reliability lever: scale-to-zero, nothing to fail at request
   time, and standalone local dev. The snapshot is (re)generated by a **CLI build step**
   that reads `ec_pv_redistributable` (reusing the warehouse plumbing) so it is
   reproducible and versioned with a warehouse build.
3. **Runs locally first; eventual target is Google Cloud Run** (scale-to-zero, portable
   container, no lock-in). The **MVP runs on localhost**; the container is kept
   cloud-agnostic. Actual cloud deploy is a later/stretch story, not MVP.
4. **MVP serves `ec_pv_redistributable` now, decoupled from E7 (hybrid).** Hybrid / flip /
   margin fields are added in a **later story once E7 lands** (E8-S8). See the
   roadmap-ordering-update note below.
5. **Redistributable-only surface from day one** ‚Äî exclude `redistributable=false` (UCSB)
   rows entirely, so the internal MVP is already the public-graduation surface (D016/D014).
6. **Location: the in-repo `usvote/api/` subpackage**, covered by the existing quality
   gates (`uv run pytest` / `ruff` / `mypy`, CI).

**Priorities driving scope/sequencing:** **cost > reliability > ease of build/maintenance**
(willing to trade a little cost for easier dev/maintenance). Traffic is initially tiny
(serving our own dashboard), but Fred may advertise to the MIT Election Lab sooner than
expected, so the foundation must be robust and public-graduation-ready even while running
local. Emphasis: **get it running quickly on a robust foundation we can grow on.**

### Roadmap ordering update (note)

The roadmap's critical path reads **"E7 + E9 ‚Üí E8."** This epic **supersedes that for the
MVP**: E8's MVP depends only on E6's `ec_pv_redistributable` view, **not** on E7 (hybrid)
or E9 (mart). Hybrid/flip/margin fields join the API in E8-S8 after E7 lands; the mart
(E9) may later become an additional read source but is not an MVP dependency. This note
should be reflected back into `docs/ROADMAP.md` (and recorded as D029).

### Scope (MVP)

- A CLI **snapshot-build** step that materializes `ec_pv_redistributable` into a versioned,
  read-only embedded snapshot (E8-S1).
- A **FastAPI app** in `usvote/api/` that loads the snapshot read-only at startup, with no
  live-DB dependency (E8-S2).
- **Core read endpoints** over the joined dataset ‚Äî by year, by state, by candidate, and a
  national/summary view ‚Äî with Pydantic response models (E8-S3).
- **OpenAPI/Swagger polish** (descriptions, examples, tags, provenance/licensing note) as a
  public-ready asset (E8-S4).
- **Tests** (unit + a snapshot-fixture integration test) mirroring the repo test layout (E8-S5).
- A **cloud-agnostic Dockerfile** ‚Äî the local-runnable containerized deliverable (E8-S6).

### Out of scope (MVP) / deferred

- **Cloud Run deploy** (E8-S7, later/stretch) ‚Äî the container is built cloud-agnostic in
  E8-S6, but standing up the cloud service, CDN, custom domain, and any auth is post-MVP.
- **Hybrid / flip / margin fields** (E8-S8, gated on E7) ‚Äî not in the first surface.
- **UCSB / `redistributable=false` rows** ‚Äî excluded from every surface by construction (D016).
- **Any write path / mutations** ‚Äî the API is strictly read-only.
- **Frontend / presentation platform** ‚Äî deferred (D001); the API's consumer is our app, but
  the app itself is not built here.
- **Live Postgres serving** ‚Äî deliberately rejected in favor of the embedded snapshot (D028).

### Success Criteria

- [ ] `ec_pv_redistributable` is materialized to a versioned read-only snapshot by a
      reproducible CLI build step that reuses the warehouse plumbing
- [ ] A FastAPI app serves the joined dataset from that snapshot with **no live-DB
      dependency** ‚Äî it starts and answers requests with Postgres stopped
- [ ] Core endpoints return correct EC+PV data by year / state / candidate and a

- [ ] Auto-generated OpenAPI/Swagger docs are complete enough to hand to an external

- [ ] **No `redistributable=false` row can appear on any endpoint** ‚Äî guarded by a test
- [ ] Unit + snapshot-fixture integration tests pass under the existing gates
      (`uv run pytest`, `ruff`, `mypy`), and CI runs them without a live DB
- [ ] `docker run` serves the API locally from a cloud-agnostic image bundling the snapshot
- [ ] The app powers our app's data needs (the D002 MVP bar)

### Stories

- [ ] #N ‚Äî E8-S1: Snapshot-build CLI + serving-store schema/contract
- [ ] #N ‚Äî E8-S2: FastAPI app skeleton + config (read-only snapshot, no live DB)
- [ ] #N ‚Äî E8-S3: Core read endpoints over `ec_pv_redistributable` (+ Pydantic models)
- [ ] #N ‚Äî E8-S4: OpenAPI/Swagger polish + provenance/licensing surface note
- [ ] #N ‚Äî E8-S5: Tests ‚Äî unit + snapshot-fixture integration, mirroring the test layout
- [ ] #N ‚Äî E8-S6: Cloud-agnostic Dockerfile (local-runnable containerized deliverable)
- [ ] #N ‚Äî E8-S7 (later/stretch): Cloud Run deploy + CDN caching
- [ ] #N ‚Äî E8-S8 (later, gated on E7): Add hybrid / flip / margin fields to the API

### Proposed endpoint / resource shape (for architect review)

Grain of `ec_pv_redistributable` is `(year, state, candidate)`, EC-left, with
`national_electoral_votes` as a per-`(year, candidate)` window sum and PV attached where
redistributable (MIT ‚Üí 1976‚Äì2024). The redistributable window is therefore **1976‚Äì2024**;
earlier years are honestly absent from this surface (that is the point ‚Äî D005/D016). A
proposed REST shape, deliberately leaving implementation open where the architect should weigh in:

| Method + path | Returns | Key query params |
|---|---|---|
| `GET /v1/elections` | list of covered election years + per-year coverage metadata (PV source, redistributable) | ‚Äî |
| `GET /v1/elections/{year}` | one election: state-by-state EC+PV rows **plus** a national summary block | `state`, `candidate` (filters) |
| `GET /v1/elections/{year}/summary` | national per-candidate view for one year: national EC total + national PV total (the flip-relevant roll-up, **minus hybrid** until E8-S8) | ‚Äî |
| `GET /v1/states/{state}` | one state across all covered years (longitudinal) | `year_from`, `year_to` |
| `GET /v1/candidates` / `GET /v1/candidates/{id}` | candidate-centric view across years | `year` |

**Response envelope (proposed):** `{ "data": [...], "meta": { "snapshot_version", "source": "MIT", "license": "CC0-1.0", "coverage": {"year_min": 1976, "year_max": 2024}, "count" } }`.
Every response (and the OpenAPI description) carries the provenance/licensing note so the
redistributable boundary is explicit to consumers.

**Deliberately open for the architect** (see per-story ACs and the open-questions list):
API versioning (`/v1/` from day one ‚Äî recommended for clean public graduation), pagination
(may be unnecessary at low-thousands-of-rows scale ‚Äî decide, don't default), the
snapshot-freshness/versioning contract and CDN cache headers (ETag/Last-Modified/Cache-Control),
CORS origins for the dashboard, and the public **candidate identifier** (per
[`docs/canonical-keys.md`](../docs/canonical-keys.md), **do not expose `candidate_id`** ‚Äî mint a
deterministic name slug if a durable public id is needed).

### E8-S1: Snapshot-build CLI + serving-store schema/contract

**Issue title:** Build a read-only snapshot of ec_pv_redistributable via a reproducible CLI step
**Labels:** `epic:internal-api`, `infrastructure`, `priority:high`

**Body:**

### Summary

Materialize the `ec_pv_redistributable` join view into a **read-only embedded snapshot**
(SQLite file or static JSON ‚Äî architect's call) via a CLI build step that reuses the
existing warehouse plumbing (`usvote/warehouse.py` / `usvote/join.py` read seam). This
snapshot is the API's only data source (D028); Postgres never sits behind the running API.
This story defines **the snapshot contract** ‚Äî its schema/shape, its versioning metadata,
and the build command ‚Äî before any HTTP code exists, so E8-S2/S3 build against a fixed target.

### Acceptance Criteria

- A CLI entry point (subcommand-style, consistent with the D027 `__main__` convention ‚Äî
  e.g. `python -m usvote.api build-snapshot` or `python -m usvote api snapshot`, architect's
  call) reads `ec_pv_redistributable` and writes the snapshot artifact to a configured path
- The build **reuses the warehouse read plumbing** (does not hand-roll a second SQL path to
  the view) and requires the local Postgres warehouse only **at build time**, never at serve time
- The snapshot carries **version/provenance metadata**: build timestamp, a snapshot schema
  version, source row counts, coverage window (`year_min`/`year_max`), and the
  `source`/`license` provenance (MIT / CC0-1.0) ‚Äî enough for the API `meta` block and for a
  consumer to know which data version they hold
- **Redistributable-only is guaranteed at the source:** the build reads `ec_pv_redistributable`
  (defined `WHERE redistributable`, D017), and a build-time assertion confirms **zero**
  `redistributable=false` / non-MIT rows entered the snapshot
- The snapshot format is documented (a short `docs/` note or module docstring) as the
  serving contract E8-S2/S3 consume
- The build is **reproducible and idempotent** ‚Äî re-running against the same warehouse build
  produces an equivalent snapshot; it is safe to regenerate alongside a `run_warehouse` build
- Unit tests cover the shape/metadata of a snapshot built from a small **synthetic** input
  frame (no live DB); any test that builds from real Postgres is `@pytest.mark.integration`
  and excluded from CI

### Implementation Notes

- **SQLite vs. static JSON is the architect's call**, weighed on cost > reliability > ease:
  SQLite gives indexed filtering for the by-year/by-state endpoints with a stdlib driver and
  no query-time memory blow-up; static JSON is trivially CDN-cacheable and dead simple but
  pushes filtering into app code. At low-thousands-of-rows scale either is viable ‚Äî decide and
  record the rationale.
- Reuse `usvote/warehouse.py` / the `join.py` read seam rather than re-querying the view
  directly, so the snapshot stays coupled to the same view definition the warehouse builds.
- The snapshot path should be config-driven via an env var mirroring the established convention
  (`USVOTE_SHAPEFILE_PATH` / `USVOTE_MIT_CSV_PATH` / `USVOTE_UCSB_HTML_DIR`) ‚Äî propose
  `USVOTE_API_SNAPSHOT_PATH`. Unset/missing raises the typed `ConfigError`.
- Do **not** expose `candidate_id` in the snapshot's external identity (D006 /
  `docs/canonical-keys.md`): it is an internal surrogate. If the snapshot needs a candidate
  key, use the canonical `name` or a deterministic slug derived from it.

### Dependencies

- E6 / #63 (`ec_pv_redistributable` view + `usvote/warehouse.py` ‚Äî already shipped)

### E8-S2: FastAPI app skeleton + config (read-only snapshot, no live DB)

**Issue title:** Stand up the FastAPI app in usvote/api reading the read-only snapshot
**Labels:** `epic:internal-api`, `enhancement`, `priority:high`

**Body:**

### Summary

Create the `src/usvote/api/` subpackage with a FastAPI application that loads the E8-S1
snapshot **read-only at startup** and serves it with **no live-DB dependency** (D028). This
story is the app skeleton, config, and a health endpoint ‚Äî the endpoints themselves are E8-S3.

### Acceptance Criteria

- `src/usvote/api/` exists as a subpackage (D015) with a FastAPI app factory and a local
  run path (e.g. `uvicorn`), documented in the README/CLAUDE.md run section
- The app loads the snapshot from `USVOTE_API_SNAPSHOT_PATH` at startup, **read-only**;
  with Postgres **stopped**, the app still starts and answers a request (proves the no-live-DB
  invariant ‚Äî this is the core reliability/cost lever)
- A `GET /health` (and/or readiness) endpoint reports app + snapshot-loaded status and the
  snapshot version/coverage from the E8-S1 metadata
- Config is centralized (snapshot path, CORS origins, API version prefix) via the established
  env-var convention; missing snapshot raises a clear typed error at startup, not a 500 at
  request time
- **CORS** is configurable (origins for the dashboard) ‚Äî a placeholder default plus an env-var
  override; the open decision on exact origins is flagged, not hard-coded to `*` silently
- The app is added to the existing quality gates (`ruff`, `mypy`, `pytest`) and CI
- New runtime deps (`fastapi`, an ASGI server such as `uvicorn`) are added to `pyproject.toml`
  + `uv.lock`

### Implementation Notes

- Keep the data-access layer thin and swappable behind an interface (a `SnapshotRepository`
  or equivalent) so the SQLite-vs-JSON choice from E8-S1 is isolated from the endpoint code
  in E8-S3, and so a future live-DB or mart (E9) backend could slot in without rewriting routes.
- Set a versioned route prefix (`/v1`) from the start ‚Äî cheap now, and it keeps the public
  graduation (stretch) from being a breaking change. (Flagged as an open decision to confirm.)
- Emit `Cache-Control` (and ideally `ETag`/`Last-Modified` tied to the snapshot version) even
  when running locally, so the eventual Cloud Run + CDN path (E8-S7) is a config change, not a
  code change.

### Dependencies

- E8-S1 (snapshot contract + artifact)

### E8-S3: Core read endpoints over `ec_pv_redistributable`

**Issue title:** Add the core read endpoints (by year / state / candidate + national summary) with Pydantic models
**Labels:** `epic:internal-api`, `enhancement`, `priority:high`

**Body:**

### Summary

Implement the MVP read endpoints over the snapshot: the joined EC+PV data by year, by state,
by candidate, and a national/summary roll-up ‚Äî each with typed Pydantic response models and
the shared response envelope. This is the surface that powers our app.

### Acceptance Criteria

- The endpoints from the epic's proposed shape are implemented (final paths are the
  architect's call, but must cover: **list years**, **one election (state rows + national
  summary)**, **national summary for a year**, **one state across years**, and a
  **candidate-centric** view)
- Every response uses a **Pydantic response model** (typed, documented fields) and the shared
  `{ data, meta }` envelope, where `meta` carries snapshot version, source/license, coverage,
  and count
- Filtering query params work as specified (`state`, `candidate`, `year_from`/`year_to`) and
  are validated (bad params ‚Üí 422, not a 500)
- The national/summary view returns per-candidate **national EC total** (the
  `national_electoral_votes` window from the view) and **national PV total** for the year ‚Äî
  the flip-relevant roll-up, **without** hybrid/flip/margin (those are E8-S8)
- Unknown year/state/candidate returns a clean **404** with a typed error body, never a stack
  trace; a year outside the redistributable window (e.g. pre-1976) is a well-described empty/404
  case, not an error
- **No `redistributable=false` data is reachable** ‚Äî asserted by a test that no endpoint can
  return a non-MIT row (defense-in-depth on top of the E8-S1 source guarantee)
- `candidate_id` is **not** exposed as the external candidate identifier (D006 /
  `docs/canonical-keys.md`); the public key is the canonical name or a deterministic slug

### Implementation Notes

- Routes read only through the E8-S2 repository interface ‚Äî no SQL/JSON parsing in the route
  handlers.
- Pagination: at low-thousands-of-rows scale it may be unnecessary; **decide explicitly**
  (a simple `limit`/`offset` with sane caps, or none for MVP) and record it. Don't silently
  return unbounded lists if a consumer could ever hit a large one.
- Response field naming should read naturally to an external consumer (this surface is the
  public-graduation surface); avoid leaking internal column names where a clearer public name
  exists.

### Dependencies

- E8-S2

### E8-S6: Cloud-agnostic Dockerfile (local-runnable containerized deliverable)

**Issue title:** Containerize the API as a cloud-agnostic image bundling the snapshot
**Labels:** `epic:internal-api`, `infrastructure`, `priority:medium`

**Body:**

### Summary

Package the API and its snapshot into a **cloud-agnostic** container that runs locally with
`docker run` ‚Äî the MVP's runnable deliverable. The image must carry no GCP-specific
assumptions, so the eventual Cloud Run deploy (E8-S7) is a deploy step, not a rebuild.

### Acceptance Criteria

- A `Dockerfile` builds an image that runs the FastAPI app via an ASGI server and serves the
  snapshot with **no external services** (no DB, no network dependency at runtime)
- The snapshot is available to the container (bundled at build, or mounted ‚Äî architect's call,
  documented) and the app reads it via `USVOTE_API_SNAPSHOT_PATH`
- `docker run -p ...` serves the API locally; `/health`, `/docs`, and a data endpoint all respond
- The image is **cloud-agnostic**: listens on `$PORT` (Cloud-Run-compatible without being
  Cloud-Run-specific), no vendor SDKs, no baked-in secrets
- The image is reasonably small and starts fast (scale-to-zero friendly ‚Äî the cost lever); the
  build is documented in the README/CLAUDE.md
- (If cheap) a CI job builds the image to catch Dockerfile rot

### Implementation Notes

- Bundling the snapshot **into** the image keeps the running service dependency-free (best for
  cost/reliability) at the cost of rebuilding to refresh data; mounting keeps data separate but
  reintroduces a runtime dependency. Weigh against cost > reliability > ease and record the call.
- Keep the container stateless and read-only so it maps cleanly onto Cloud Run later.

### Dependencies

- E8-S3 (a serving app to containerize); E8-S1 (snapshot)
